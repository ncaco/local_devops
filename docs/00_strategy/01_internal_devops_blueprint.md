# 사내 DevOps 청사진

## 왜 이 문서를 먼저 쓰는가

지금 문제는 도구가 없는 게 아닙니다. 기준이 없습니다.

- 소스관리는 SVN
- 가상화는 Hyper-V 중심
- 빌드와 배포는 수동
- 팀은 기획, 디자인, 개발로 나뉘어 있고 약 50인 규모

이 상태에서 도구만 하나씩 추가하면 운영 복잡도만 늘어납니다. 먼저 "우리 회사의 표준 DevOps 경로"를 문서로 고정해야 합니다.

## 목표

1. 무료 오픈소스 기반으로 시작한다.
2. LTS 중심으로 운영한다.
3. 소스관리, 이미지 저장, 빌드 실행, 배포 흐름을 한 저장소 관점에서 연결한다.
4. 작은 팀이 직접 운영 가능한 수준의 복잡도로 유지한다.
5. 수동 배포를 단계적으로 제거한다.

## 비목표

- 첫 단계에서 쿠버네티스를 도입하지 않는다.
- 첫 단계에서 모든 레거시 프로젝트를 한 번에 Git으로 이전하지 않는다.
- 첫 단계에서 운영망까지 완전 자동화하지 않는다.

## 현재 상태

| 항목 | 현재 | 문제 |
|------|------|------|
| 소스관리 | SVN | 브랜치 전략, PR 문화, 자동화 연계가 약함 |
| 실행 환경 | Hyper-V | 운영은 되지만 개발 표준화가 약함 |
| 빌드 | 수동 | 사람 의존도가 높고 재현성이 낮음 |
| 배포 | 수동 | 누가 언제 무엇을 배포했는지 추적이 어려움 |
| 문서 | 메모 수준 | 표준이 아닌 개인 노하우에 가까움 |

## 원칙

### 1. 로컬에서 먼저 작동해야 한다

개발자가 자기 PC에서 재현하지 못하는 운영 구조는 오래 못 갑니다.

### 2. 한 번에 다 바꾸지 않는다

SVN, Hyper-V, 수동배포가 이미 있는 조직입니다. 이건 신설 회사가 아닙니다. 전환 계획이 있어야 합니다.

### 3. 배포보다 추적 가능성이 먼저다

자동배포가 중요한 게 아니라, 누가 어떤 버전을 어떤 경로로 올렸는지 보이는 게 먼저입니다.

### 4. 운영 인력이 적다는 전제로 설계한다

50인 회사에서 플랫폼 팀이 따로 크지 않다면, 멋있는 구조보다 덜 부서지는 구조가 이깁니다.

## 제안 아키텍처

### 선택 스택

| 영역 | 선택 | 이유 |
|------|------|------|
| 개발 표준 환경 | WSL2 + Docker Engine | Windows 기반 사내 환경과 궁합이 좋고 재현 가능성이 높음 |
| 리버스 프록시 | Caddy | 로컬/내부 TLS와 라우팅 구성이 단순함 |
| Git 서버 | Gitea | 가볍고 자가 호스팅 난이도가 낮음 |
| 컨테이너 레지스트리 | Docker Registry | 최소 구성으로 시작 가능 |
| CI 실행기 | Gitea Actions Runner 또는 범용 Runner | Git push 이후 빌드 자동화 연결점 |
| 컨테이너 운영 UI | Portainer | 초기 운영 가시성 확보 |
| 데이터베이스 | PostgreSQL | 표준성, 안정성, 운영 문서 풍부 |

### 왜 이 조합인가

이 조합의 장점은 "지금 당장 굴릴 수 있다"는 점입니다.

Git 서버, 레지스트리, 러너, 프록시가 한 묶음으로 이어집니다. 반대로 GitHub Enterprise, GitLab, Kubernetes로 바로 가면 기능은 많지만 사내 첫 표준화 문서로는 과합니다. 50인 조직에서 첫 단계는 플랫폼보다 습관 전환이 더 큰 일입니다.

## 시스템 흐름

```text
개발자
  |
  v
WSL2 + Docker 로컬 개발
  |
  v
Gitea 저장소
  |
  +--> Gitea Runner
  |      |
  |      v
  |   Build / Test / Image Push
  |      |
  |      v
  |   Docker Registry
  |
  v
배포 대상 서버
  |
  v
Caddy Reverse Proxy
  |
  +--> 내부 전용 관리 서비스
  |      - Portainer
  |      - Gitea
  |      - Registry
  |      - Runner
  |
  +--> 개발 배포 영역
  |
  +--> 운영 배포 영역
```

## 네트워크/접근 정책

### 내부 전용

- Portainer
- Gitea
- Registry
- Runner

### 내부/외부 혼용

- 개발 배포 환경
- 별도 DNS, IP, SSL 정책 적용
- 포트는 `2xxxx` 대역처럼 역할별 규칙을 둔다

### 외부 전용

- 운영 배포 환경
- 별도 DNS, IP, SSL 정책 적용
- 포트는 `3xxxx` 대역처럼 운영 대역을 분리한다

## 의사결정 로그

### 결정 1. Git 표준은 Gitea로 시작한다

- 이유: SVN 기반 조직이 Git 문화를 도입할 때 가장 먼저 필요한 건 가벼운 저장소, PR, 권한관리, CI 연결점이다.
- 포기한 것: 대규모 올인원 플랫폼의 깊은 기능.

### 결정 2. CI는 "자동 빌드"부터 시작한다

- 이유: 테스트, 패키징, 이미지 푸시까지 자동화되면 수동배포 절반은 이미 끝난다.
- 포기한 것: 첫 단계의 완전 자동 운영 배포.

### 결정 3. 배포는 개발/운영을 분리한다

- 이유: 같은 툴을 써도 배포 등급은 달라야 한다. 개발 편의성과 운영 안정성을 같은 규칙으로 묶으면 둘 다 망가진다.

### 결정 4. WSL 문서는 개별 설치문서가 아니라 표준 환경 문서의 일부다

- 이유: WSL은 목표가 아니라 기반이다. 설치순서만 적으면 왜 필요한지 금방 잊힌다.

## 단계별 전환 계획

### Phase 1. 기반 고정

- WSL2, Docker, Caddy, Portainer, PostgreSQL, Gitea, Registry, Runner 설치
- 내부 DNS 및 로컬 도메인 규칙 확정
- 팀 공용 설치 문서 작성

### Phase 2. Git 전환 시작

- 신규 프로젝트는 Git 우선
- 기존 SVN 프로젝트는 우선순위별 미러링 또는 단계 전환
- 브랜치 전략과 PR 규칙 정의

### Phase 3. 빌드 자동화

- Gitea push 기준 빌드 파이프라인 생성
- 테스트, 이미지 빌드, 레지스트리 푸시 자동화
- 실패 로그 확인 경로 고정

### Phase 4. 배포 자동화

- 개발 환경부터 자동 배포
- 운영은 승인 단계 포함
- 롤백 절차를 문서화하고 훈련

## 운영 표준

### 브랜치/배포 기준

- `main`: 배포 가능 상태 유지
- 기능 브랜치: 작업 단위 분리
- 태그: 운영 배포 기준점

### 이미지 태깅 기준

- `app:branch-buildnumber`
- `app:release-YYYYMMDD-N`
- 운영 배포는 커밋 SHA 또는 릴리스 태그를 반드시 남긴다

### 로그/추적 기준

- 누가 배포했는지 남는다
- 어떤 이미지가 올라갔는지 남는다
- 실패한 빌드 로그를 다시 볼 수 있다

이 세 가지가 안 되면 자동화가 아니라 블랙박스입니다.

## 리스크

| 리스크 | 설명 | 대응 |
|--------|------|------|
| SVN 잔존 | 모든 프로젝트가 한 번에 이동되지 않음 | Git 우선 신규 적용, 기존은 단계 전환 |
| 운영 지식 편중 | 특정 인원만 설치/복구 가능 | 설치 문서와 장애 복구 절차 분리 작성 |
| 포트 충돌 | 로컬/내부 서비스가 계속 늘어남 | 포트 대역 규칙 문서화 |
| 인증서 관리 | 내부/외부 혼용 환경에서 SSL 정책이 복잡해짐 | Caddy 정책과 인증서 발급 범위 분리 |

## 추천 결론

이 프로젝트의 첫 승리는 "도구 설치 완료"가 아닙니다.

"신규 프로젝트는 Git + 자동 빌드 + 이미지 저장까지 표준 경로로 간다"를 만드는 겁니다.

그게 되면 수동 운영이 줄기 시작합니다. 그 전에는 그냥 서버가 몇 개 더 생긴 겁니다.
