# SNS 자동화 배포 서비스 정보구조(IA) 문서

작성일: 2026-03-29
작성 방식: IA/UX 서브에이전트 초안 기반 정리
기준 문서: `AGENT.md`, `01_sns_automation_deployment_service_plan.md`, `02_prd.md`, `04_technical_design.md`, `11_design_strategy.md`

## 1. 문서 목적

이 문서는 SNS 자동화 배포 서비스의 화면 구조와 탐색 흐름을 고정하기 위한 정보구조 문서다.

목표는 운영자가 콘텐츠 작성, 승인, 예약, 게시, 실패 복구, 감사 추적까지 하나의 흐름으로 끊기지 않고 이동하게 만드는 것이다.

이 문서는 다음을 결정한다.

- 사용자가 처음 들어왔을 때 무엇을 먼저 보게 할지
- 어떤 역할이 어떤 화면에 접근하는지
- 화면이 어떤 순서로 연결되는지
- 빈 상태와 오류 상태를 어디서 어떻게 보여줄지
- 이후 채널 확대나 권한 확장 시 구조를 어떻게 유지할지

## 2. 사용자 역할별 IA

### 관리자

조직 전체를 운영하는 역할이다.

조직 설정, 브랜드 관리, SNS 계정 연결, 사용자 권한 관리, 감사로그 조회까지 접근한다.

### 운영자

콘텐츠를 작성하고 예약하고 실패를 복구하는 역할이다.

대시보드, 콘텐츠 작성, 게시 작업 상세, 실패 콘솔, 예약 목록을 중심으로 사용한다.

### 승인자

게시 전 검토와 승인을 담당하는 역할이다.

승인함, 콘텐츠 상세, 예약 정보, 반려 사유 입력 화면을 주로 사용한다.

### 조회자

운영 현황을 보기만 하는 역할이다.

대시보드 요약, 배포 이력, 감사로그, 실패 현황 일부를 확인한다.

## 3. 글로벌 네비게이션

글로벌 네비게이션은 "조직과 브랜드를 먼저 고르고, 그 안에서 상태를 본다"는 원칙으로 설계한다.

### 기본 구성

- Overview
- Content Pipeline
- Approvals
- Scheduled Posts
- Failures
- Channel Accounts
- Audit Log
- Settings

### 상단 공통 요소

- 조직 선택기
- 브랜드 선택기
- 상태 요약 배지
- 알림 아이콘
- 사용자 메뉴

### 내비게이션 원칙

- 전역 필터는 조직과 브랜드다.
- 목록보다 상태 요약이 먼저 보여야 한다.
- 설정은 항상 마지막에 둔다.
- 운영자가 가장 자주 보는 것은 Overview, Approvals, Failures다.

## 4. 주요 화면군 구조

### 4.1 Overview

운영 상태를 한눈에 보는 관제 화면이다.

오늘 예정 게시, 승인 대기, 실패 작업, 계정 이상 상태를 요약한다.

### 4.2 Content Pipeline

콘텐츠 작성과 검증, 예약 전 상태를 다루는 화면군이다.

초안 작성, 채널별 검증, 첨부 에셋 확인, 예약 설정이 이 안에 포함된다.

### 4.3 Approvals

승인자가 검토하고 반려하는 화면군이다.

승인 대기 목록, 승인 상세, 반려 사유 기록, 승인 이력이 포함된다.

### 4.4 Scheduled Posts

예약된 게시와 게시 예정 작업을 보는 화면군이다.

예약 시간, 대상 채널, 게시 상태, 실행 전 점검 정보를 제공한다.

### 4.5 Failures

실패한 게시를 복구하는 화면군이다.

실패 사유, 재시도 가능 여부, 영향 범위, 재실행 이력을 중심으로 본다.

### 4.6 Channel Accounts

SNS 계정 연결과 상태를 관리하는 화면군이다.

연결 상태, 토큰 만료, 권한 부족, 재인증 흐름을 제공한다.

### 4.7 Audit Log

누가 무엇을 언제 바꿨는지 추적하는 화면군이다.

필터, 검색, 상세 이력, 상태 변경 로그를 제공한다.

### 4.8 Settings

조직, 역할, 알림, 정책, 기본값을 관리하는 화면군이다.

주 업무 흐름과 분리해 마지막에 둔다.

## 5. 핵심 사용자 플로우

### 플로우 A. 콘텐츠 작성에서 게시까지

```text
Overview
  -> Content Pipeline
  -> Content Detail
  -> Approval Request
  -> Approval Result
  -> Scheduled Posts
  -> Publish Job Detail
```

### 플로우 B. 승인 처리

```text
Approvals
  -> Approval Detail
  -> 승인 또는 반려
  -> 결과 반영
  -> Audit Log 기록
```

### 플로우 C. 실패 복구

```text
Failures
  -> Failure Detail
  -> 원인 확인
  -> 재시도 또는 수정
  -> Publish Job Detail
  -> 상태 갱신
```

### 플로우 D. 계정 이상 대응

```text
Channel Accounts
  -> Account Detail
  -> 토큰 이상 확인
  -> 재인증 또는 중단
  -> 관련 작업 차단
```

## 6. 화면 간 전이 구조

### 전이 원칙

- Overview는 모든 화면의 출발점이다.
- 목록 화면에서 상세 화면으로 들어가고, 상세 화면에서 관련 이력으로 이동한다.
- 실패 화면은 게시 작업 상세와 반드시 연결된다.
- 승인 화면은 콘텐츠 상세와 예약 정보로 연결된다.

### 주요 전이

- Overview -> Content Pipeline
- Overview -> Approvals
- Overview -> Failures
- Content Pipeline -> Approval Detail
- Approval Detail -> Scheduled Posts
- Scheduled Posts -> Publish Job Detail
- Publish Job Detail -> Failures
- Failures -> Publish Job Detail
- Channel Accounts -> Account Detail -> Re-authentication
- Audit Log -> 관련 엔티티 상세

### 전이 설계 포인트

- 상세 화면에서 다른 화면으로 이동할 때 맥락을 잃지 않도록 상단에 현재 조직, 브랜드, 상태를 고정 표시한다.
- 승인과 게시는 분리된 단계지만, 사용자는 한 흐름으로 느껴야 한다.
- 실패 복구는 단순 이동이 아니라 상태 복원이다. 복구 전후 상태가 비교 가능해야 한다.

## 7. 빈 상태/오류 상태 설계 포인트

### 빈 상태

- 단순히 "데이터 없음"만 보여주지 않는다.
- 다음 행동을 함께 제시한다.
- 예: 채널 계정이 없으면 "계정 연결하기", 승인 대기 0건이면 "최근 게시 확인하기"를 제공한다.

### 오류 상태

- 오류는 숨기지 않고, 무엇이 실패했는지 먼저 보여준다.
- 사용자가 바로 행동할 수 있도록 복구 버튼 또는 재시도 버튼을 제공한다.
- 기술 오류 메시지는 운영 언어로 번역한다.

### 화면별 포인트

- Overview: 부분 실패와 지연 상태를 별도로 노출한다.
- Content Pipeline: 검증 실패는 필드 단위로 표시한다.
- Approvals: 반려 사유 입력을 강제한다.
- Failures: 재시도 가능 여부와 불가 사유를 분리한다.
- Channel Accounts: 연결 실패와 토큰 만료를 구분한다.
- Audit Log: 조회 실패 시 필터 초기화 또는 범위 축소 안내를 제공한다.

## 8. 향후 확장 고려사항

### 채널 확장

채널이 늘어나도 IA는 바뀌지 않아야 한다.

채널은 콘텐츠와 계정의 속성으로 흡수하고, 메뉴 구조는 유지한다.

### 권한 확장

역할이 늘어나더라도 역할별로 메뉴를 숨기고 보여주는 방식으로 처리한다.

화면을 새로 쪼개지 말고 접근 권한으로 분리한다.

### 멀티브랜드 확장

브랜드 수가 늘어날수록 전역 필터와 최근 사용 브랜드가 중요해진다.

브랜드 전환 비용이 낮아야 운영이 무너지지 않는다.

### 운영 확장

감사로그, 실패 복구, 알림, 정책 관리가 커질수록 Settings와 Audit Log의 정보 밀도가 높아질 수 있다.

이때도 주 업무 화면과 관리 화면의 경계는 유지한다.

### 디자인 확장

향후 대시보드와 실패 콘솔이 복잡해져도, 상태 중심 구조와 정보 우선순위는 유지해야 한다.

예쁜 구조보다 운영자가 매일 쓰는 구조가 먼저다.

## 최종 구조 요약

```text
Overview
  -> Content Pipeline
  -> Approvals
  -> Scheduled Posts
  -> Failures
  -> Channel Accounts
  -> Audit Log
  -> Settings
```

이 IA의 핵심은 화면 수를 늘리는 것이 아니라, 운영 흐름을 끊지 않는 것이다.

사용자는 콘텐츠를 만들고, 승인하고, 예약하고, 게시하고, 실패를 복구하는 과정을 하나의 시스템으로 경험해야 한다.
