# SNS 자동화 배포 서비스 개발 전략 문서

작성일: 2026-03-29
작성 방식: Eng 전략 서브에이전트 초안 기반 정리
기준 문서: `AGENT.md`, `01_sns_automation_deployment_service_plan.md`, `02_prd.md`, `04_technical_design.md`

## 1. 개발 원칙

이 서비스는 CRUD 백오피스가 아니라, SNS 콘텐츠를 안전하게 배포하는 운영 시스템으로 만든다. 화면보다 상태, 상태보다 작업, 작업보다 복구 가능성이 중요하다.

- 승인 없는 게시를 기본적으로 막는다.
- 외부 SNS API 실패는 숨기지 않고 표준 에러로 변환한다.
- 모든 배포 작업은 idempotent 하게 설계한다.
- 채널별 차이는 코어 로직이 아니라 어댑터 계층에서 흡수한다.
- 예약, 승인, 게시, 재시도, 감사로그는 하나의 흐름으로 이어져야 한다.
- 운영자가 사람 손으로 복구할 수 있어야 한다. 자동화만 있고 복구가 없으면 사고다.
- 작은 MVP라도 관측성, 보안, 테스트는 생략하지 않는다.
- 구현은 최소 diff를 유지하되, 경계는 명확하게 자른다.

## 2. 시스템 경계와 모듈 전략

권장 아키텍처는 Next.js 프론트엔드, FastAPI 백엔드, PostgreSQL, 비동기 게시 워커 구조다. 핵심은 "작성"과 "배포"를 분리하고, 배포는 오케스트레이션으로 다루는 것이다.

### 2.1 서비스 경계

- `frontend`: 운영자용 대시보드, 콘텐츠 작성, 승인, 이력 조회
- `api`: 인증, 권한, 도메인 규칙, 조회/명령 API
- `domain`: 콘텐츠, 승인, 게시, 채널, 감사로그의 핵심 비즈니스 규칙
- `worker`: 예약 시각 기반 게시 실행, 재시도, 결과 저장
- `adapter`: Instagram, X, Facebook, YouTube 등 채널별 외부 API 연동
- `db`: PostgreSQL 영속화 계층
- `ops`: 알림, 로그, 메트릭, 장애 대응 도구

### 2.2 모듈 전략

백엔드는 도메인별로 나눈다. 공통 유틸이 먼저 커지면 망한다. 먼저 상태가 있는 모델을 잡고, 그 다음 서비스와 워커를 붙인다.

- `content`: 초안, 변형 콘텐츠, 검증 규칙
- `approval`: 승인 요청, 승인, 반려, 이력
- `publish`: 게시 작업, 재시도, 결과 저장
- `channel`: 채널 계정, 토큰 상태, 채널별 제약
- `audit`: 누가 언제 무엇을 했는지 기록
- `notification`: 실패, 만료, 큐 적체 알림
- `shared`: 진짜 공통만 둔다. 과한 공통화 금지

### 2.3 데이터 모델 전략

- `Organization`, `Brand`, `User`, `Role`
- `ChannelAccount`
- `ContentItem`, `ContentVariant`
- `ApprovalRequest`
- `PublishJob`, `PublishAttempt`
- `AuditLog`
- `Notification`

상태 전환은 문자열 나열이 아니라 명시적 상태 머신으로 관리한다. `DRAFT -> PENDING_APPROVAL -> APPROVED -> SCHEDULED -> PUBLISHING -> PUBLISHED/FAILED` 같은 전환을 코드와 테스트에서 고정한다.

## 3. 단계별 구현 순서

순서는 사용자 가치가 아니라 위험 순으로 간다. 가장 먼저 불안정한 부분은 외부 SNS 연동이 아니라, 내부 상태 전환과 권한이다.

### Phase 1. 기반 고정

- 인증/권한 모델 구현
- 조직/브랜드/사용자 경계 구현
- 콘텐츠, 승인, 게시 작업의 기본 스키마 구현
- 감사로그 저장 구조 구현
- Next.js FSD 기본 구조 세팅
- FastAPI 프로젝트 구조와 `uv` 기반 의존성 관리 세팅

### Phase 2. 핵심 워크플로우

- 콘텐츠 작성
- 승인 요청, 승인, 반려
- 예약 게시 생성
- 게시 상태 조회
- 실패 이력 저장
- 재시도 액션

이 단계에서 반드시 "승인 전 게시 금지"가 강제되어야 한다. UI에서 막는 것만으로는 부족하다. 백엔드가 막아야 한다.

### Phase 3. 채널 어댑터와 워커

- 채널 어댑터 인터페이스 정의
- 1차 채널 2종 구현
- 게시 워커 구현
- 재시도 정책 구현
- rate limit, timeout, auth failure 표준화
- idempotency key 적용

### Phase 4. 운영 화면과 알림

- 운영 대시보드
- 실패 작업 목록
- 채널 계정 건강 상태
- 승인 대기함
- 알림 채널 연결
- 감사 추적 화면

### Phase 5. 확장과 안정화

- 추가 채널 확대
- 플랫폼별 콘텐츠 변형
- 리포트
- 베스트 타임 추천
- 운영 도구 강화

## 4. 테스트 전략

테스트는 "잘 되는지"보다 "망가질 때 안전한지"를 본다. SNS 배포 서비스는 실패 테스트가 곧 제품 품질이다.

### 4.1 테스트 계층

- 단위 테스트: 상태 전환, 검증 규칙, 에러 매핑, 재시도 정책
- 통합 테스트: API와 DB, 승인 흐름, 게시 작업 생성
- 워커 테스트: 예약 시각 도달, 중복 실행 방지, 실패 복구
- E2E 테스트: 작성 -> 승인 -> 예약 -> 게시 -> 실패 -> 재시도
- 계약 테스트: 채널 어댑터의 입력/출력 계약
- 회귀 테스트: 토큰 만료, rate limit, 네트워크 timeout

### 4.2 반드시 있어야 할 시나리오

- 승인 없는 게시가 차단되는지
- 같은 작업이 두 번 실행돼도 중복 게시가 안 되는지
- 외부 API 429, 5xx, timeout이 각각 다른 처리로 가는지
- 토큰 만료 시 후속 작업이 멈추는지
- 예약 작업이 유실되지 않는지
- 실패 후 재시도가 최대 횟수를 넘기면 수동 조치로 전환되는지
- nil, empty, 과대 길이 입력이 모두 검증되는지

### 4.3 테스트 우선순위

- 우선순위 1: 상태 전환과 승인 통제
- 우선순위 2: 게시 작업과 재시도
- 우선순위 3: 채널 어댑터
- 우선순위 4: 운영 UI와 조회

테스트는 많이, 하지만 산만하게 말고 핵심 경로에 두껍게 넣는다. 이 서비스는 "한 번 사고 나면 큰일 나는" 종류다.

## 5. 배포 및 운영 전략

개발, 검증, 운영 환경을 명확하게 분리한다. 포트와 인프라도 역할별로 일관되게 유지한다.

### 5.1 환경 전략

- 로컬: 개발자 생산성 우선, Docker PostgreSQL 포함
- 개발: Vercel + Railway로 통합 검증
- 운영: AWS 기반으로 안정성, 보안, 확장성 확보

### 5.2 배포 순서

- DB 스키마는 backward-compatible 하게 먼저 배포
- 백엔드 API 배포
- 워커 배포
- 프론트엔드 배포
- 알림과 운영 대시보드 연결
- smoke test 수행

### 5.3 롤백 전략

- 기능 플래그를 먼저 둔다
- DB 롤백이 어려운 변경은 expand-only로 설계한다
- 외부 연동 장애가 나면 어댑터를 차단하고 내부 상태를 보존한다
- 배포 후 문제 발생 시 마지막 안전 버전으로 되돌릴 수 있어야 한다

### 5.4 운영 체크포인트

- 큐 적체
- 실패율 급증
- 토큰 만료 증가
- 예약 누락
- 재시도 실패
- 특정 채널만 연속 실패

이 서비스는 배포가 끝이 아니다. 배포 이후에 얼마나 빨리 실패를 보느냐가 제품이다.

## 6. 보안과 관측성

보안과 관측성은 나중에 붙이는 장식이 아니다. SNS 자동화는 토큰과 권한을 다루는 순간 바로 운영 리스크가 된다.

### 6.1 보안 원칙

- SNS 토큰은 암호화 저장
- 역할 기반 접근 제어 적용
- 조직/브랜드 단위 데이터 격리
- 승인자와 운영자 권한 분리
- 모든 민감 행위는 감사로그 남김
- 입력 검증은 백엔드에서 강제
- 외부 API 호출 파라미터는 화이트리스트 기준으로 제한

### 6.2 관측성 원칙

- 요청 단위 trace id 부여
- 게시 작업 id와 외부 채널명을 로그에 남김
- 실패 코드와 정규화 메시지 기록
- 채널별 성공률 메트릭 수집
- 큐 길이, 재시도 횟수, 실패 유형별 건수 측정
- 토큰 만료, 실패 임계치, 큐 적체 알림

### 6.3 운영자가 봐야 할 것

- 지금 예약된 게시가 몇 개인지
- 오늘 실패한 작업이 무엇인지
- 어느 채널이 문제인지
- 승인 대기 중인 항목이 무엇인지
- 재시도 가능한 실패인지 아닌지

로그가 없으면 운영은 감이다. 감으로 굴리는 자동화는 그냥 사고 예약 시스템이다.

## 7. 팀 운영 방식

이 프로젝트는 frontend, backend, ops, design이 분리되지 않게 돌아가야 한다. 각자 따로 잘하면 전체는 망한다.

- 제품/기획은 승인 기준과 운영 시나리오를 먼저 고정한다.
- 백엔드는 상태 머신과 데이터 모델을 먼저 잠근다.
- 프론트엔드는 화면보다 상태와 실패 케이스를 먼저 설계한다.
- QA는 happy path가 아니라 failure path를 먼저 본다.
- 운영은 장애 대응 runbook을 구현 전부터 준비한다.
- 변경은 작게, 하지만 문맥은 충분히 공유한다.

권장 작업 방식은 "작은 PR, 명확한 책임, 높은 검증 밀도"다. 한 PR에 화면, DB, 워커, 어댑터를 다 넣지 않는다. 유지보수는 항상 그 대가를 청구한다.

## 8. 주요 기술 리스크와 대응

### 리스크 1. 외부 SNS API 정책 변경

- 문제: API 제약, 권한, 호출 정책이 바뀔 수 있다.
- 대응: 채널 어댑터 분리, 계약 테스트, 실패 표준화

### 리스크 2. 토큰 만료와 권한 박탈

- 문제: 계정은 살아있어 보여도 실제 게시가 실패할 수 있다.
- 대응: 계정 health check, 만료 사전 탐지, 후속 작업 자동 차단

### 리스크 3. 중복 게시

- 문제: 예약 작업이 두 번 실행되면 사고다.
- 대응: idempotency key, 작업 상태 잠금, 중복 실행 차단

### 리스크 4. 승인 누락

- 문제: 사람이 실수하면 바로 잘못된 게시가 나간다.
- 대응: 서버 단의 승인 게이트, 상태 전환 검증, UI와 API 이중 차단

### 리스크 5. 큐 적체와 지연

- 문제: 작업이 밀리면 예약 시각이 깨진다.
- 대응: 큐 모니터링, 우선순위 조정, 작업 분산, 알림

### 리스크 6. 운영 가시성 부족

- 문제: 실패 원인을 재현 못 하면 지원비용이 커진다.
- 대응: 작업 이력, 실패 사유, trace id, 감사로그, 운영 대시보드

### 리스크 7. 범위 확장

- 문제: 생성 AI, 분석, 리스닝까지 욕심내면 코어가 흐려진다.
- 대응: 배포 신뢰성을 1차 목표로 고정하고, 확장은 2차 이후로 둔다

## 결론

이 서비스의 개발 전략은 분명해야 한다. 먼저 배포 신뢰성, 그 다음 운영성, 마지막에 확장성이다.

순서를 뒤집으면 데모는 예쁘지만 제품은 약해진다. 이 프로젝트는 SNS를 "올리는 도구"가 아니라 "안전하게 배포하는 시스템"으로 만들어야 한다. 그게 개발 전략의 중심이다.
