# SNS 자동화 배포 서비스 PRD

작성일: 2026-03-29
연관 문서: `01_sns_automation_deployment_service_plan.md`

## 1. 제품 개요

SNS 자동화 배포 서비스는 여러 SNS 채널에 콘텐츠를 안전하게 배포하기 위한 운영형 서비스다.

이 제품의 목적은 콘텐츠 생성 자체가 아니라, 승인된 콘텐츠가 적절한 시점에 정확한 채널로 게시되고, 실패 시 즉시 복구 가능한 상태를 만드는 것이다.

## 2. 문제 정의

현재 SNS 운영은 다음 문제를 반복해서 만든다.

- 여러 브랜드와 계정을 동시에 운영할 때 실수 가능성이 커진다.
- 승인 없는 게시나 잘못된 계정 게시가 사고로 이어진다.
- 채널별 제약이 달라 운영 규칙을 사람 기억에 의존하게 된다.
- 예약은 되었지만 실제 게시 실패를 뒤늦게 발견하는 경우가 많다.
- 실패 원인, 승인자, 게시 이력을 사후 추적하기 어렵다.

## 3. 목표

### 3.1 제품 목표

- 승인 기반 SNS 게시 파이프라인을 표준화한다.
- 예약 게시 성공률을 높인다.
- 실패 탐지와 복구 시간을 줄인다.
- 운영자의 수동 확인 작업을 줄인다.

### 3.2 비즈니스 목표

- 한 명의 운영자가 더 많은 브랜드와 채널을 관리할 수 있게 한다.
- SNS 운영 사고를 줄여 신뢰 비용을 낮춘다.
- 장기적으로 대행사형 멀티테넌트 SaaS로 확장 가능한 기반을 만든다.

## 4. 비목표

- AI 기반 고급 콘텐츠 생성 엔진
- 자체 영상/이미지 편집 스튜디오
- 광고 집행 관리
- 소셜 리스닝 전반
- 고급 분석 BI 제품

## 5. 타깃 사용자

### 5.1 1차 타깃

- 사내 SNS 운영 담당자
- 중소형 브랜드 마케팅 팀
- 승인 체계가 필요한 콘텐츠 운영 조직

### 5.2 2차 타깃

- 멀티 브랜드를 관리하는 대행사
- 복수 담당자와 승인자가 분리된 팀

## 6. 핵심 사용자 시나리오

### 시나리오 A. 운영자가 게시를 준비한다

1. 운영자가 브랜드와 채널을 선택한다.
2. 게시할 콘텐츠를 작성한다.
3. 채널별 검증 규칙을 통과한다.
4. 예약 시각을 지정하고 승인 요청을 보낸다.

성공 기준:

- 규칙 위반 콘텐츠는 예약 전에 차단된다.
- 승인 요청이 명확히 생성된다.

### 시나리오 B. 승인자가 검토한다

1. 승인자는 승인 대기 목록을 확인한다.
2. 콘텐츠와 게시 채널, 예약 시각을 검토한다.
3. 승인 또는 반려한다.

성공 기준:

- 승인 전에는 게시 작업이 실행되지 않는다.
- 승인/반려 이력과 담당자가 기록된다.

### 시나리오 C. 시스템이 예약 게시를 수행한다

1. 예약 시각이 되면 게시 작업이 생성된다.
2. 채널 어댑터가 외부 SNS API를 호출한다.
3. 성공 또는 실패 결과가 저장된다.

성공 기준:

- 성공 여부가 지연 없이 반영된다.
- 실패 시 사유가 사람이 이해할 수 있는 수준으로 표준화된다.

### 시나리오 D. 운영자가 실패를 복구한다

1. 실패 목록에서 원인을 확인한다.
2. 자동 재시도 또는 수동 수정 후 재실행한다.
3. 최종 결과를 확인한다.

성공 기준:

- 같은 실패를 반복 재현하지 않도록 시도 이력이 남는다.
- 운영자는 "왜 실패했는지"를 바로 알 수 있다.

## 7. 핵심 기능 요구사항

### 7.1 계정 및 권한

- 조직/브랜드 단위 분리
- 운영자/승인자/관리자 역할 분리
- SNS 채널 계정 연결 및 상태 확인
- 토큰 만료, 권한 이상 상태 감지

### 7.2 콘텐츠 관리

- 콘텐츠 초안 작성
- 미디어 첨부
- 예약 시각 설정
- 채널별 제약 검증
- 플랫폼별 변형 콘텐츠 저장

### 7.3 승인 워크플로우

- 승인 요청 생성
- 승인/반려 처리
- 반려 사유 기록
- 승인 전 게시 차단

### 7.4 게시 파이프라인

- 예약 시점 기반 작업 생성
- 작업 큐 적재
- 채널 어댑터 실행
- 결과 상태 갱신
- 자동 재시도 정책 적용

### 7.5 운영 가시성

- 오늘 게시 예정 목록
- 실패 작업 목록
- 채널별 성공률
- 최근 승인 및 반려 이력
- 게시 시도 상세 이력

## 10. 상태 모델

### 콘텐츠 상태

```text
DRAFT
  -> PENDING_APPROVAL
  -> REJECTED
  -> APPROVED
  -> SCHEDULED
  -> PUBLISHING
  -> PUBLISHED
  -> FAILED
```

### 게시 작업 상태

```text
QUEUED
  -> RUNNING
  -> SUCCEEDED
  -> FAILED
  -> RETRY_SCHEDULED
  -> CANCELLED
```

## 11. KPI

### 운영 KPI

- 예약 게시 성공률 98% 이상
- 실패 탐지 평균 시간 5분 이내
- 승인 요청부터 게시 완료까지 평균 리드타임 감소
- 자동 재시도 후 회복률

### 제품 KPI

- 주간 활성 운영자 수
- 조직당 월간 게시 건수
- 실패 작업 수동 처리 비율 감소

## 13. 운영 정책

- 승인 없는 자동 게시 금지
- 채널 토큰 이상 시 즉시 배포 중단
- 동일 콘텐츠 중복 게시 방지
- 게시 실패는 자동 재시도 횟수 제한 적용
- 모든 상태 변경은 감사로그 저장

## 14. 수용 기준

### MVP 수용 기준

- 운영자가 콘텐츠를 작성해 승인 요청할 수 있다.
- 승인자가 승인하면 예약 시점에 게시 작업이 실행된다.
- 실패 시 원인과 재시도 상태를 확인할 수 있다.
- 게시/승인/실패 이력이 저장된다.
- 관리자 대시보드에서 운영 상태를 볼 수 있다.

|--------|------|------|
