# SNS 자동화 배포 서비스 기획서

작성일: 2026-03-29
기준 문서: `AGENT.md`
문서 성격: 산출물 기준 제품/기술 기획 초안

## 1. 한 줄 정의

SNS 자동화 배포 서비스는 여러 SNS 채널에 대해 콘텐츠 검수, 예약, 게시, 실패 복구, 이력 추적을 하나의 파이프라인으로 제공하는 운영형 배포 플랫폼이다.

핵심은 "콘텐츠를 만드는 도구"가 아니라 "문제 없이 배포되게 만드는 도구"다.

## 2. 왜 이걸 만들어야 하나

기존 SNS 도구는 대체로 일정 관리와 발행 UI에 집중한다.

실제 운영팀이 힘든 지점은 따로 있다.

- 여러 브랜드와 여러 채널을 동시에 관리해야 한다.
- 승인 없이 잘못 발행되면 사고가 난다.
- 플랫폼별 제약이 달라 같은 콘텐츠라도 배포 규칙이 다르다.
- 예약은 되었는데 실제 게시 실패를 늦게 발견하면 기회비용이 크다.
- 누가 무엇을 언제 승인했고 왜 실패했는지 추적이 어렵다.

즉, 사용자가 원하는 것은 "게시 기능"이 아니라 "운영 신뢰성"이다.

## 3. 제품 가설

### 3.1 타깃 사용자

- 사내 마케팅 운영팀
- 여러 브랜드를 동시에 관리하는 대행사 운영팀
- 승인 체계가 필요한 중소형 조직

### 3.2 Job To Be Done

"여러 SNS 계정에 콘텐츠를 안전하게 배포하고, 실패를 즉시 감지하고, 누가 승인했고 어떤 결과가 났는지 남기고 싶다."

### 3.3 차별화 포인트

- 채널별 게시 성공률과 실패 사유를 표준화해서 보여준다.
- 예약 발행보다 한 단계 앞선 "배포 파이프라인" 개념으로 설계한다.
- 초안, 승인, 예약, 게시, 재시도, 감사로그까지 하나의 흐름으로 묶는다.
- 운영자가 "왜 실패했는지"를 바로 이해할 수 있게 한다.

## 4. 제품 원칙

- 배포 실패는 숨기지 않는다.
- 승인 없는 게시를 기본적으로 막는다.
- 채널별 차이는 UI가 아니라 정책 계층에서 흡수한다.
- 수동 운영보다 자동화가 우선이지만, 언제든 사람이 개입할 수 있어야 한다.
- MVP라도 감사로그, 재시도, 상태 추적은 빠지면 안 된다.

## 5. 권장 제품 범위

이번 기획은 그린필드 기능이므로 `SCOPE EXPANSION` 기준으로 보되, 실제 1차 개발 범위는 "운영 가능한 최소 플랫폼"으로 자른다.

### 5.1 MVP 범위

- 조직, 브랜드, SNS 채널 계정 연결 관리
- 콘텐츠 초안 작성 및 예약 발행
- 승인 요청, 승인, 반려 워크플로우
- 채널별 게시 작업 큐
- 게시 상태 추적: 대기, 승인대기, 예약, 게시중, 성공, 실패, 재시도대기
- 실패 사유 표준화 및 재시도
- 배포 이력 조회
- 관리자용 운영 대시보드

### 5.2 2차 확장 범위

- AI 기반 플랫폼별 문안 변환
- 이미지/영상 에셋 검수 체크리스트
- 베스트 타임 추천
- 댓글/반응 수집과 간단 리포트
- 다국어 캠페인 템플릿

### 5.3 제외 범위

- 자체 이미지/영상 편집기
- 고급 소셜 리스닝
- CRM 전체 기능
- 광고 집행 플랫폼

## 6. 구현 접근안 비교

### 접근안 A. 배포 파이프라인 중심 MVP, 권장

- 요약: 콘텐츠를 공통 스키마로 저장하고, 채널별 어댑터가 최종 게시를 수행한다.
- 장점: 구조가 명확하고 운영 신뢰성에 집중하기 쉽다.
- 단점: 채널별 어댑터 설계가 초기부터 필요하다.
- 적합성: 현재 요구사항과 가장 잘 맞는다.

### 접근안 B. 채널별 기능 개별 구현

- 요약: SNS별로 화면, 정책, 배포 로직을 각각 만든다.
- 장점: 초기 1개 채널 대응은 빠를 수 있다.
- 단점: 채널이 늘수록 코드와 운영 복잡도가 급격히 증가한다.
- 적합성: 단기 데모에는 가능하지만 제품화에는 불리하다.

### 접근안 C. 콘텐츠 생성 중심 서비스

- 요약: AI 문안 생성과 추천을 전면에 두고 배포는 보조 기능으로 둔다.
- 장점: 데모는 화려하다.
- 단점: 실제 운영팀의 핵심 pain point인 승인, 실패 복구, 감사 추적을 약하게 만든다.
- 적합성: 현 시점 우선순위가 아니다.

### 결론

접근안 A를 채택한다.

이 서비스의 승부처는 생성이 아니라 배포 신뢰성이다.

## 8. 핵심 도메인 모델

- Organization: 서비스 이용 조직
- Brand: 조직 내 운영 브랜드
- ChannelAccount: SNS 연결 계정
- ContentItem: 게시 대상 콘텐츠 원본
- ContentVariant: 플랫폼별 변형 콘텐츠
- ApprovalRequest: 승인 요청 단위
- PublishJob: 실제 게시 작업
- PublishAttempt: 게시 시도 이력
- DeploymentPolicy: 채널별 게시 규칙
- AuditLog: 행위 및 상태 변경 기록

## 9. 핵심 사용자 흐름

### 9.1 기본 흐름

```text
초안 작성
  -> 검수 요청

  -> 예약 확정
  -> 게시 작업 생성
  -> 채널별 게시 실행
  -> 성공/실패 기록
  -> 재시도 또는 종료
```

### 9.2 실패 복구 흐름

```text

  -> 자동 재시도 가능 여부 판단
  -> 자동 재시도 또는 수동 수정 요청
  -> 최종 상태 확정
```

## 18. CEO 관점 결론

이 서비스는 "SNS 관리 툴"이 아니라 "SNS 배포 운영 시스템"으로 정의해야 한다.

그래야 제품 우선순위가 흔들리지 않는다.

1차에서 반드시 잡아야 할 것은 예쁜 캘린더보다 승인 통제, 게시 신뢰성, 실패 복구, 감사 추적이다.

이 네 가지가 잡히면 이후 AI 문안 생성, 추천, 분석은 얹기 쉽다.

반대로 이 네 가지 없이 AI부터 올리면 데모는 되지만 운영 제품은 안 된다.
