# SNS 자동화 배포 서비스 테스트 전략

작성일: 2026-03-29
작성 방식: QA/테스트 전략 서브에이전트 초안 기반 정리
기준 문서: `02_prd.md`, `04_technical_design.md`, `10_engineering_strategy.md`, `05_api_spec.md`, `06_database_design.md`, `11_design_strategy.md`

## 1. 문서 목적

이 문서는 SNS 자동화 배포 서비스의 품질 기준과 검증 범위를 고정하기 위한 테스트 전략 문서다.

핵심 목적은 승인, 예약, 게시, 실패 복구, 감사 추적이 실제 운영에서 깨지지 않도록 사전에 검증하는 것이다.

이 문서는 다음을 정의한다.

- 무엇을 반드시 테스트해야 하는지
- 어떤 수준의 테스트가 어디까지 책임지는지
- 실패와 에지케이스를 어디까지 커버할지
- 릴리즈 전 통과 기준을 무엇으로 둘지

## 2. 테스트 원칙

- 승인 없는 게시가 절대 통과되지 않도록 테스트한다.
- 실패 경로를 happy path와 같은 비중으로 검증한다.
- 외부 SNS API는 항상 불안정하다고 가정하고 테스트한다.
- 재시도, 중복 실행, 순서 뒤바뀜, 지연 도착을 기본 전제로 둔다.
- DB 상태가 최종 진실이며, UI 표시만으로 완료를 판단하지 않는다.
- 테스트는 적게 넓게가 아니라, 핵심 경로를 깊게 커버하는 방향으로 작성한다.
- 실패는 숨기지 말고, 기대한 실패인지 아닌지를 분명히 분리한다.
- 비동기 작업은 한 번만 실행된다고 믿지 않는다. idempotency를 테스트한다.

## 3. 테스트 피라미드

### Unit

도메인 규칙, 상태 전환, 검증 로직, 에러 매핑, 재시도 정책을 검증한다.

가장 많이 작성하고 가장 빨리 돌아가야 한다.

### Integration

API와 DB, 승인 흐름, 게시 작업 생성, 채널 어댑터 계약을 검증한다.

상태 저장과 조회의 일관성을 본다.

### E2E

사용자 여정을 통째로 검증한다.

콘텐츠 작성 -> 승인 요청 -> 승인 -> 예약 -> 게시 -> 실패 -> 재시도 흐름이 실제로 연결되는지 확인한다.

### Manual/Exploratory

대시보드, 실패 콘솔, 승인 화면, 모바일 반응형 같은 UX 민감 영역은 탐색 테스트를 추가한다.

예쁜 화면이 아니라 운영자가 실수 없이 움직일 수 있는지 본다.

## 4. 기능별 테스트 범위

### 인증/권한

- 로그인, 토큰 갱신, 로그아웃이 정상 동작하는지
- 역할별 접근 제어가 정확한지
- 다른 조직 데이터 접근이 차단되는지

### 조직/브랜드/채널 계정

- 조직과 브랜드의 계층 관계가 유지되는지
- 채널 계정 연결, 재인증, 해제가 올바른 상태로 전환되는지
- 토큰 만료와 권한 부족이 올바른 상태로 표시되는지

### 콘텐츠

- 초안 생성, 수정, 삭제, 검증이 정상 동작하는지
- 채널별 길이, 금지 요소, 예약 시각 검증이 수행되는지
- 상태별 수정 가능 범위가 지켜지는지

### 승인

- 승인 요청 생성이 중복되지 않는지
- 승인, 반려, 보류가 상태 전환 규칙을 지키는지
- 반려 사유가 필수로 저장되는지

### 게시 작업

- 예약 시각 도달 시 작업이 생성되는지
- 게시 작업 상태가 queued/running/succeeded/failed로 정확히 이동하는지
- 재시도와 취소가 상태 규칙을 위반하지 않는지

### 대시보드/운영 화면

- 집계 수치가 실제 DB 상태와 일치하는지
- 실패 목록, 채널 건강 상태, 최근 작업 타임라인이 정확한지
- 빈 상태와 오류 상태가 의도한 메시지를 보여주는지

## 5. 실패/에지케이스 테스트

- nil/empty 입력이 모두 검증되는지
- 예약 시각이 과거인 경우 차단되는지
- 승인 전에 게시 시도가 들어오면 막히는지
- 같은 idempotency key로 중복 요청을 보내도 중복 게시가 안 되는지
- 네트워크 timeout, 429, 5xx, malformed response가 각각 다른 경로로 처리되는지
- 토큰이 만료된 채널 계정에 게시 작업이 생성되지 않는지
- 작업이 실행 중일 때 사용자가 새로고침하거나 뒤로 가도 상태가 깨지지 않는지
- 한 콘텐츠에 여러 채널이 붙었을 때 일부만 실패하는 partial failure가 올바르게 표시되는지
- 승인자와 운영자가 같은 작업을 동시에 건드렸을 때 충돌이 안전하게 처리되는지

## 6. 비동기/재시도 테스트

- 예약 작업이 한 번만 큐에 들어가는지
- 워커가 죽었다가 살아나도 작업이 유실되지 않는지
- 동일 작업이 두 번 실행되어도 중복 게시가 발생하지 않는지
- 재시도 횟수 제한이 지켜지는지
- 지수 백오프가 기대한 간격으로 적용되는지
- dead-letter 또는 수동 조치 상태로 전환되는지
- 이벤트 순서가 뒤집혀도 최종 상태가 DB 기준으로 수렴하는지
- 웹훅 또는 비동기 이벤트가 at-least-once로 와도 중복 처리에 안전한지

## 7. 보안/권한 테스트

- 조직 경계를 넘어선 조회와 수정이 차단되는지
- admin, operator, approver, viewer 역할이 명세대로 제한되는지
- 다른 브랜드의 콘텐츠, 승인, 게시 작업을 우회 접근할 수 없는지
- 채널 토큰이 평문으로 노출되지 않는지
- 감사로그가 민감 정보 없이도 추적 가능한지
- 입력값에 대한 길이 초과, 문자열 주입, JSON 변조 시도가 차단되는지
- OAuth callback의 state 검증 실패가 거부되는지
- 파괴적 액션이 권한 없이 실행되지 않는지

## 8. 성능/회귀 테스트

- 대시보드 요약이 실제 운영 데이터 규모에서 느려지지 않는지
- publish_jobs, approval_requests, audit_logs 조회에 인덱스가 제대로 타는지
- 예약 작업이 몰릴 때 큐 적체가 허용 범위 안인지
- 대용량 이력에서 페이지네이션이 안정적으로 동작하는지
- 채널별 실패가 늘었을 때 API 응답 시간이 급격히 악화되지 않는지
- 이전에 막았던 버그가 다시 살아나지 않는지 회귀 테스트를 유지하는지
- 릴리즈마다 승인, 재시도, 중복 실행, 토큰 만료 케이스를 고정 세트로 재실행하는지

## 9. 릴리즈 게이트와 QA 체크리스트

### 릴리즈 게이트

- 승인 없는 게시 차단 테스트 통과
- 중복 게시 차단 테스트 통과
- 토큰 만료 및 권한 부족 테스트 통과
- 실패 복구와 재시도 테스트 통과
- 대시보드 집계 정확성 통과
- 보안/권한 테스트 통과
- 핵심 E2E 시나리오 통과

### QA 체크리스트

- 콘텐츠 작성, 승인, 예약, 게시, 실패 복구가 실제로 한 흐름으로 이어지는가
- 실패 시 운영자가 다음 행동을 바로 알 수 있는가
- 모바일에서 최소한 상태 확인과 승인/반려가 가능한가
- 빈 화면이 단순 공백이 아니라 다음 행동을 제안하는가
- 로그와 감사 추적으로 3주 뒤에도 사건을 재구성할 수 있는가
- 운영자가 새로 들어와도 10분 안에 시스템 상태를 이해할 수 있는가

이 서비스는 데모가 아니라 운영 도구다.

따라서 테스트의 기준도 "보인다"가 아니라 "안전하게 운영된다"여야 한다.
