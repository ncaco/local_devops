# SNS 자동화 배포 서비스 디자인 전략

작성일: 2026-03-29
작성 방식: Design 전략 서브에이전트 초안 기반 정리
기준 문서: `AGENT.md`, `01_sns_automation_deployment_service_plan.md`, `02_prd.md`, `04_technical_design.md`

## 1. 디자인 목표

이 제품의 디자인은 "예쁜 관리 화면"이 아니라 "운영자가 실수 없이 배포를 끝내는 도구"를 만드는 데 맞춘다. 핵심 목표는 세 가지다.

- 상태를 즉시 읽게 만든다. 지금 무엇이 대기 중이고, 무엇이 승인됐고, 무엇이 실패했는지 한눈에 보여야 한다.
- 실수를 막는다. 승인 전 게시, 잘못된 브랜드 선택, 중복 배포 같은 사고를 인터페이스 수준에서 줄인다.
- 복구를 빠르게 만든다. 실패했을 때 원인, 영향 범위, 다음 행동이 바로 보여야 한다.

이 서비스의 디자인은 "콘텐츠 제작"보다 "배포 통제"에 가까워야 한다. 감성보다 정확성, 장식보다 가독성, 탐색보다 즉시성. 운영자는 예쁜 화면이 아니라, 현재 상황을 바로 이해할 수 있는 화면이 필요하다.

성공 기준은 단순하다. 사용자가 30초 안에 현재 운영 상태를 파악하고, 3클릭 안에 승인하거나 실패를 복구할 수 있어야 한다.

## 2. 핵심 사용자 여정

### 2.1 계정 연결과 첫 설정

처음 사용자는 조직과 브랜드를 만들고, SNS 계정을 연결한다. 이 여정의 핵심은 "복잡한 설정"이 아니라 "연결이 정상인지 확신하는 경험"이다.

- 첫 화면에서 해야 할 일은 명확해야 한다.
- 연결 상태, 토큰 유효성, 게시 가능 여부를 즉시 보여줘야 한다.
- 실패한 연결은 숨기지 말고, 왜 실패했는지 설명해야 한다.

### 2.2 콘텐츠 작성과 검수

운영자는 콘텐츠를 작성하고, 채널별 규칙을 확인한 뒤 승인 요청을 보낸다.

- 작성 화면은 단순한 에디터가 아니라 게시 전 검증 허브여야 한다.
- 길이 초과, 미디어 규격 위반, 채널 미지원 요소는 입력 시점에 바로 드러나야 한다.
- 사용자는 "저장됨"보다 "게시 가능한 상태인지"를 더 중요하게 본다.

### 2.3 승인과 예약

승인자는 대기 목록에서 콘텐츠를 검토하고 승인 또는 반려한다.

- 승인 화면은 정보가 많아도 흐름이 단순해야 한다.
- 게시 대상 채널, 예약 시각, 변형 문구, 첨부 에셋, 반려 이력을 한 묶음으로 보여줘야 한다.
- 승인 버튼은 크더라도 가볍게 보여주면 안 된다. 사고가 나는 지점이다.

### 2.4 게시 실행과 실패 복구

예약 시각이 되면 게시가 실행되고, 실패 시 운영자가 즉시 원인을 확인한다.

- 성공은 조용하지만 명확해야 한다.
- 실패는 큰 경고보다 먼저 원인을 설명해야 한다.
- 재시도는 무조건 가능한 버튼이 아니라, 왜 재시도 가능한지 판단 정보와 함께 보여줘야 한다.

### 2.5 운영 대시보드 점검

운영자는 하루에 몇 번 메인 대시보드와 실패 목록을 본다.

- 오늘 예정 게시, 지연 작업, 실패 작업, 계정 이상 상태가 가장 먼저 보여야 한다.
- 이 화면은 제품의 "관제실"이다.
- 숫자가 많더라도 압박감이 아니라 통제감을 줘야 한다.

## 3. 정보구조

정보구조는 "브랜드 단위 운영"을 중심으로 잡는다. 글로벌 내비게이션보다 먼저 보여야 하는 것은 현재 어떤 조직, 어떤 브랜드, 어떤 배포 상태인지다.

### 권장 1차 구조

- Overview
- Content Pipeline
- Approvals
- Scheduled Posts
- Failures
- Channel Accounts
- Audit Log
- Settings

### 계층 원칙

- 전역 필터는 조직과 브랜드다.
- 목록은 상태 중심으로 정렬한다.
- 상세는 콘텐츠, 승인, 게시, 실패 이력을 한 화면에서 이어서 본다.
- 관리 기능은 계정 설정보다 뒤로 둔다.

### 정보 우선순위

1. 현재 운영 상태
2. 승인 대기와 예약 대기
3. 실패와 재시도
4. 계정 연결 상태
5. 이력과 감사로그

이 서비스는 검색 중심 제품이 아니다. 사용자는 찾아다니기보다 "지금 뭘 봐야 하는지"를 바로 알아야 한다.

## 4. 화면군 전략

### 4.1 Overview Dashboard

가장 중요한 화면이다. 운영자는 여기서 오늘의 배포 상태를 읽는다.

- 상단에는 브랜드 선택, 다음 게시, 실패 건수, 승인 대기 건수를 둔다.
- 중앙에는 상태별 카드와 최근 작업 타임라인을 배치한다.
- 하단에는 실패 목록과 계정 이상 경고를 둔다.

이 화면의 역할은 요약이다. 작업을 끝내는 곳이 아니라, 무엇을 들어가서 처리할지 알려주는 곳이다.

### 4.2 Content Composer

콘텐츠 작성과 검증이 결합된 화면이다.

- 왼쪽은 작성, 오른쪽은 채널별 검증 결과처럼 분리된 시각 구조가 적합하다.
- 플랫폼별 글자 수, 금지 요소, 미디어 규격, 예약 시각을 즉시 확인할 수 있어야 한다.
- 저장보다 게시 가능성에 초점을 둔다.

### 4.3 Approval Inbox

승인자는 여기서 판단한다.

- 각 카드에 브랜드, 예약 시각, 채널, 작성자, 위험 신호를 요약해서 보여준다.
- 승인과 반려는 같은 화면에서 처리하되, 반려 사유는 필수다.
- 승인 시에는 영향 범위와 예약 시각을 다시 보여준다.

### 4.4 Publish Job Detail

실행 단위의 진실이 있는 화면이다.

- 하나의 게시 작업에 대해 생성, 승인, 예약, 실행, 결과, 재시도 이력을 한 번에 본다.
- 상태 변화를 시간순으로 보여주는 타임라인이 핵심이다.
- 운영자가 실패 원인과 담당자를 바로 파악할 수 있어야 한다.

### 4.5 Failure Console

실패를 해결하는 전용 화면이다.

- 실패 사유를 기술 용어가 아니라 운영 언어로 번역해 보여준다.
- 재시도 가능 여부, 차단 원인, 다음 권장 행동을 함께 보여준다.
- 여러 실패를 한 번에 처리하는 배치 액션은 있되, 남용되면 안 된다.

### 4.6 Channel Accounts

계정 연결과 건강 상태를 관리한다.

- 연결 상태, 토큰 만료, 권한 부족, 비정상 응답을 구분해서 보여준다.
- 정상보다 이상 상태를 더 눈에 띄게 둔다.
- 연결 재인증은 짧고 확실하게 끝나야 한다.

### 4.7 Audit Log

감사로그는 운영 신뢰의 증거다.

- 누가, 언제, 무엇을 바꿨는지, 결과는 어땠는지 남긴다.
- 이력은 숨김이 아니라 제품 기능이다.
- 검색과 필터는 강해야 하지만, 화면은 복잡해지지 않아야 한다.

## 5. 상태 설계

이 제품은 상태가 곧 UX다. 로딩, 빈 상태, 오류, 성공이 모두 다르게 설계되어야 한다.

| 상태 | 표현 | 사용자 기대 | 설계 원칙 |
|---|---|---|---|
| Loading | 스켈레톤, 자리 고정, 이전 데이터 유지 | 기다리는 동안 맥락을 잃지 않기 | 스피너만 던지지 말 것 |
| Empty | 첫 행동 CTA, 예시, 가이드 문구 | "아직 없다"가 아니라 "다음에 무엇을 해야 하는지" | 빈 화면은 시작 화면이어야 함 |
| Error | 페이지 상단 경고 + 인라인 복구 버튼 | 실패 이유와 해결 경로 | 숨기지 말고 고쳐야 함 |
| Success | 짧은 토스트 + 화면 내 결과 반영 | 방금 일이 끝났음을 확신 | 과한 축하 금지 |
| Partial | 일부 성공, 일부 실패를 분리 표시 | 전체가 아닌 부분 상태를 인지 | 운영 도구답게 섬세해야 함 |

### 화면별 상태 포인트

- Overview는 부분 실패와 지연 상태를 반드시 보여줘야 한다.
- Approval Inbox는 승인 대기 0건을 허전하게 만들지 말고, 다음 예약이나 실패 복구로 연결해야 한다.
- Failure Console은 오류를 무감각하게 만들면 안 된다. 심각도와 조치 가능성을 분리해서 보여줘야 한다.

## 6. 컴포넌트/디자인시스템 방향

이 제품의 미감은 "조용한 관제실" 쪽이 맞다. 장식적인 소비자 앱처럼 보이면 신뢰가 떨어진다.

### 톤 앤 매너

- 차분하고 명확한 UI
- 높은 정보 밀도
- 상태 중심 시각 언어
- 불필요한 애니메이션 최소화
- 강한 대비와 명확한 계층

### 타이포그래피

- 본문은 한국어 가독성이 좋은 산세리프 계열을 사용한다.
- 숫자와 시간, 상태 코드에는 고정폭 또는 숫자 정렬이 좋은 서브 스타일을 둔다.
- 제목은 크기보다 무게로 계층을 만든다.

### 컬러

- 기본 배경은 밝거나 너무 화려하지 않은 중립 톤
- 주요 액션은 신뢰감 있는 블루 계열
- 성공, 경고, 실패는 상태를 빠르게 읽게 하는 보조 색으로 사용
- 빨강은 많이 쓰지 말고, 정말 위험한 상태에만 쓴다

### 주요 컴포넌트

- App Shell
- Brand Switcher
- Status Chip
- Approval Card
- Job Timeline
- Failure Panel
- Account Health Badge
- Audit Row
- Filter Bar
- Sticky Action Bar
- Inline Validation Message

### 상호작용 원칙

- 상태 변경은 즉시 반영하되, 되돌릴 수 있는 경우를 명확히 보여준다.
- 파괴적 액션은 항상 맥락과 함께 확인시킨다.
- 버튼은 많아도 되지만, 의미가 겹치면 안 된다.
- 한 화면에서 하나의 주 행동만 강하게 보여준다.

### 모션

- 페이지 진입, 상태 전환, 성공/실패 반영에만 짧은 모션을 쓴다.
- 장식용 움직임은 뺀다.
- 운영자는 예쁜 전환보다 반응 속도와 안정감을 먼저 느껴야 한다.

## 7. 모바일/반응형 원칙

모바일은 이 제품의 주 사용 환경이 아니다. 그래도 무시하면 안 된다. 운영자는 이동 중에 승인, 경고 확인, 실패 대응을 볼 수 있어야 한다.

### 모바일 우선 범위

- 대시보드 요약
- 승인 대기 확인
- 실패 알림 확인
- 계정 이상 상태 확인
- 간단 승인/반려

### 모바일에서 피해야 할 것

- 긴 작성 폼
- 복잡한 테이블 편집
- 대량 배치 작업
- 다중 패널 비교 화면

### 반응형 구조 원칙

- 데스크톱에서는 좌우 분할, 모바일에서는 단일 열
- 표는 카드형 요약으로 전환
- 사이드 패널은 바텀시트로 바뀔 수 있어야 함
- 주요 액션은 하단 고정 버튼으로 유지
- 긴 텍스트는 접기/펼치기로 처리

### 화면 우선순위

1. 상태 확인
2. 승인 또는 반려
3. 실패 원인 확인
4. 상세 편집은 데스크톱에서

이 제품의 모바일은 "작업 도구"라기보다 "상태 확인과 빠른 판단 도구"에 가깝다.

## 8. 신뢰를 만드는 UX 원칙

이 제품은 신뢰가 곧 기능이다. 사용자가 시스템을 믿지 못하면 자동화는 의미가 없다.

- 상태를 숨기지 않는다. 실패, 지연, 부분 성공을 모두 드러낸다.
- 누가 했는지 남긴다. 승인자, 변경자, 실행자를 항상 추적 가능하게 한다.
- 행동의 결과를 바로 보여준다. 버튼을 눌렀는데 무슨 일이 일어났는지 모르면 안 된다.
- 용어를 운영 언어로 번역한다. 기술 오류 메시지를 그대로 보여주지 않는다.
- 승인과 게시를 분리한다. 사람의 개입 지점을 명확히 둬야 사고가 줄어든다.
- 브랜드와 채널 컨텍스트를 항상 보여준다. 잘못된 계정에 게시하는 사고를 막는다.
- 되돌릴 수 있는 행동과 되돌릴 수 없는 행동을 분명히 구분한다.
- 감사로그는 내부용이 아니라 사용자 신뢰용이다.

이 제품의 UX는 "빠르게 끝내는 것"보다 "안전하게 끝내는 것"이 우선이다. 운영 화면은 사용자를 놀라게 하면 안 된다. 대신, 지금 무슨 일이 일어나고 있는지 한 번에 이해되게 해야 한다. 그것이 이 서비스의 디자인 전략이다.
