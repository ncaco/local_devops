# SNS 자동화 배포 서비스 배포/운영 전략

작성일: 2026-03-29
작성 방식: 운영/배포 전략 서브에이전트 초안 기반 정리
기준 문서: `AGENT.md`, `01_sns_automation_deployment_service_plan.md`, `04_technical_design.md`, `10_engineering_strategy.md`, `05_api_spec.md`, `06_database_design.md`

## 1. 문서 목적

이 문서는 SNS 자동화 배포 서비스의 배포 방식, 운영 기준, 장애 대응, 롤백 기준을 고정한다.

목표는 개발, 검증, 운영 환경이 서로 다르게 움직이더라도 배포 사고를 줄이고, 장애가 나도 빠르게 원인을 찾고 되돌릴 수 있게 만드는 것이다.

이 서비스는 게시 시스템이다. 배포가 곧 운영 신뢰다.

그래서 이 문서는 "어떻게 올릴까"보다 "올린 뒤 어떻게 안 터지게 할까"에 초점을 둔다.

## 2. 환경별 배포 전략

### Local

로컬은 개발자 작업용이다.

Frontend는 로컬 포트 `13000`, Backend는 `14000`, DB는 `15432` 계열을 사용한다.

- Frontend는 `Next.js` 개발 서버로 띄운다.
- Backend는 `uv` 기반 FastAPI를 로컬에서 실행한다.
- DB는 Docker PostgreSQL로 띄운다.
- 외부 SNS API는 실제 연동보다 목(mock) 또는 샌드박스 우선으로 검증한다.
- 로컬은 기능 개발과 실패 시나리오 재현이 목적이다.

### Dev

개발 환경은 팀 통합 검증용이다.

Frontend는 Vercel, Backend와 DB는 Railway를 우선으로 둔다. 포트는 `23000`, `24000`, `25432` 계열로 분리한다.

- PR 단위로 통합 검증한다.
- 실제 OAuth, 게시 작업, 재시도 흐름을 확인한다.
- 운영 데이터는 사용하지 않는다.
- Dev는 "배포 전에 깨질 걸 먼저 깨보는 곳"이다.

### Prod

운영 환경은 AWS 기반으로 고정한다.

Frontend, Backend, DB 모두 운영 기준의 격리와 접근 통제를 적용한다. 포트는 `33000`, `34000`, `35432` 계열로 본다.

- 배포는 자동화하되 승인 게이트를 둔다.
- 운영 DB는 백업과 복구 절차가 먼저다.
- 장애 시 운영자 개입 없이도 최소 기능이 유지돼야 한다.
- Prod는 실사용이기 때문에 실험을 하지 않는다.

## 3. 인프라 구성 원칙

- Frontend, Backend, Worker, DB는 역할별로 분리한다.
- 배포 단위는 작게 가져간다. 한 번에 다 바꾸지 않는다.
- DB 마이그레이션은 backward-compatible 하게 설계한다.
- 게시 작업은 비동기 Worker로 처리하고, API 요청과 분리한다.
- 외부 SNS 어댑터는 코어와 격리한다.
- 환경별 설정은 코드에 넣지 않고 환경 변수와 비밀 저장소로 분리한다.
- 모든 배포는 idempotent 하게 재실행 가능해야 한다.
- 운영 트래픽과 관리 트래픽을 논리적으로 분리한다.
- 실패 복구 경로를 배포 구조 안에 포함한다. 나중에 붙이면 늦다.

## 4. 배포 절차

1. DB 마이그레이션을 먼저 배포한다. 기존 코드와 호환돼야 한다.
2. Backend API를 배포한다.
3. Worker를 배포한다.
4. Frontend를 배포한다.
5. 알림, 웹훅, 대시보드 연결을 확인한다.
6. 운영 체크리스트로 smoke test를 수행한다.

배포 전 확인 항목:

- 마이그레이션이 롤백 가능하거나 최소한 forward-only로 안전한지
- 새 API가 이전 API와 충돌하지 않는지
- Worker가 중복 실행돼도 중복 게시하지 않는지
- 설정값이 환경별로 제대로 주입되는지

배포 후 5분 확인 항목:

- 로그인 가능 여부
- 대시보드 응답 여부
- 승인 요청 생성 여부
- 예약 게시 생성 여부
- Worker 큐 적재 여부
- 에러 로그 급증 여부

## 5. 롤백 전략

롤백은 앱, Worker, DB를 분리해서 생각한다.

한 번에 다 되돌리는 방식은 위험하다.

- 기능 문제는 feature flag로 먼저 차단한다.
- Frontend 문제는 이전 빌드로 되돌린다.
- Backend 문제는 직전 안정 버전으로 되돌린다.
- Worker 문제는 즉시 중지하고 재개 전까지 큐를 보존한다.
- DB 문제는 backward-compatible migration 원칙 때문에 보통 "되돌리기"보다 "보완 마이그레이션"으로 푼다.
- 외부 SNS 장애는 내부 롤백이 아니라 어댑터 차단과 재시도 중지로 대응한다.

롤백 판단 기준:

- 승인 없는 게시 가능성이 생기면 즉시 차단한다.
- 중복 게시 가능성이 생기면 즉시 Worker를 내린다.
- 데이터 무결성이 깨지면 배포보다 복구를 먼저 한다.

## 6. 모니터링/알림/런북

### 모니터링

- 게시 성공률
- 게시 실패율
- 재시도 회복률
- 승인 대기 시간
- 큐 적체 시간
- 토큰 만료 건수
- 채널별 오류 비율
- DB 연결 실패 수
- 5xx 비율과 p95 응답시간

### 알림

- 채널 계정 토큰 만료 임박
- 게시 실패 임계치 초과
- 큐 backlog 급증
- DB 연결 실패
- 특정 채널 연속 실패
- 승인 플로우 차단
- 비정상 재시도 반복

### 런북

- 실패 작업은 publish job id 기준으로 추적한다.
- 같은 채널에서 연속 실패가 나면 해당 채널 어댑터를 먼저 의심한다.
- 토큰 만료는 재인증이 우선이다.
- DB 이상은 앱 재시도보다 연결 상태와 쿼리 지연을 먼저 본다.
- 큐 적체는 Worker 수와 스케줄 간격을 먼저 조정한다.
- 운영자는 "어디서 멈췄는지"를 먼저 찾고, 그 다음 원인을 찾는다.

## 7. 장애 시나리오와 대응

### SNS API rate limit

- 증상: 게시 실패, 429 증가, 재시도 폭증
- 대응: 해당 채널 어댑터 재시도 중지, 지수 백오프 적용, 운영 알림 발송

### 토큰 만료

- 증상: 인증 실패, 채널 계정 unhealthy 상태
- 대응: 계정 연결 차단, 재인증 유도, 관련 작업 보류

### 중복 게시

- 증상: 동일 콘텐츠가 두 번 발행될 위험
- 대응: `idempotency_key` 확인, Worker 중복 실행 차단, 작업 상태 잠금 확인

### DB 마이그레이션 실패

- 증상: API 시작 실패, 조회 오류, Worker 중단
- 대응: 마이그레이션 즉시 중단, 이전 버전 복구 또는 보완 마이그레이션 수행

### 큐 적체

- 증상: 예약 시각 지연, 대기 증가
- 대응: Worker 스케일 조정, 작업 우선순위 재검토, 대기열 알림 강화

### 비밀정보 유출 의심

- 증상: 토큰 노출, 이상한 인증 시도
- 대응: 즉시 비밀 회전, 관련 세션 폐기, 로그 범위 점검, 계정 연결 재검증

## 8. 운영 체크리스트

배포 전:

- [ ] 마이그레이션이 안전한가
- [ ] 환경 변수와 비밀값이 준비됐는가
- [ ] Worker와 API 버전이 맞는가
- [ ] 알림 채널이 연결됐는가
- [ ] 롤백 기준이 정해졌는가

배포 직후:

- [ ] 로그인과 권한 확인
- [ ] 대시보드 응답 확인
- [ ] 승인 요청 생성 확인
- [ ] 예약 게시 생성 확인
- [ ] Worker 큐 적재 확인
- [ ] 에러 로그 확인

일상 운영:

- [ ] 실패 작업 목록 확인
- [ ] 토큰 만료 알림 확인
- [ ] 큐 backlog 확인
- [ ] 채널별 오류 비율 확인
- [ ] 감사로그 확인

## 9. 보안/비밀정보 운영 원칙

- 비밀정보는 코드 저장소에 두지 않는다.
- 토큰, client secret, webhook secret은 환경 변수 또는 비밀 저장소로만 관리한다.
- 운영, 개발, 로컬의 비밀값은 서로 재사용하지 않는다.
- 로그에 access token, refresh token, secret을 남기지 않는다.
- 비밀정보는 정기적으로 회전한다.
- 채널 계정 권한은 최소 권한 원칙으로 준다.
- 감사로그는 남기되 민감값은 마스킹한다.
- 운영자는 필요한 범위만 접근한다.
- 비밀정보 변경 시 관련 Worker와 API를 순차적으로 재기동한다.
- 유출 가능성이 있으면 배포보다 비밀 회전이 먼저다.

이 서비스의 운영은 "배포 성공"으로 끝나지 않는다.

배포 뒤에 게시가 안정적으로 돌아가고, 실패가 보이고, 되돌릴 수 있어야 운영이 된다.
