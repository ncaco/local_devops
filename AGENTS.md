# Repository Guidelines

## 프로젝트 구조 및 모듈 구성
현재 이 저장소에는 Git 메타데이터만 있으며, 실제 프로젝트 코드는 아직 없습니다. 코드가 추가되면 아래 구조를 기본으로 유지하세요.

- `src/`: 애플리케이션 또는 자동화 코드
- `tests/`: `src/`와 대응되는 자동화 테스트
- `scripts/`: 로컬 개발 및 CI 보조 스크립트
- `docs/`: 설계 문서, 운영 가이드, 아키텍처 메모

모듈은 작고 명확하게 나누고, 기능이나 책임 단위로 파일을 묶는 것을 권장합니다.

## 빌드, 테스트, 개발 명령어
현재는 프로젝트 전용 빌드 스크립트가 없습니다. 새 도구를 추가할 때는 실행 진입점을 문서화하고, 명령어 이름을 일관되게 유지하세요.

- `git status`: 작업 트리 상태 확인
- `git log --oneline -5`: 최근 커밋 메시지 형식 확인
- 향후 `npm test`, `make test`, `pytest` 같은 실제 명령어를 이 문서에 추가

패키지 매니저나 태스크 러너를 도입했다면, 같은 변경에 이 문서도 함께 갱신하세요.

## 코딩 스타일 및 네이밍 규칙
도입한 언어의 포매터와 린터 규칙을 우선 따르세요. 기준이 아직 없다면 아래 원칙을 적용합니다.

- 텍스트 파일은 UTF-8 사용
- 들여쓰기는 언어 관례가 없다면 공백 4칸 사용
- 스크립트와 파일 이름은 `snake_case` 사용
- `deploy_checks.py`, `sync_state.ts`처럼 역할이 드러나는 이름 사용
- 함수는 작게 유지하고, 부수 효과는 명확하게 드러낼 것

수동 스타일 논쟁보다 자동 포맷팅 도구를 우선합니다.

## 테스트 가이드
새 기능을 추가할 때는 테스트도 함께 추가하세요. 실행 가능한 테스트 경로 없이 저장소가 커지지 않도록 유지해야 합니다.

- 테스트는 `tests/` 아래에 배치
- 가능하면 소스 파일 이름을 반영해 작성
- 예시: `tests/test_deploy_checks.py`
- 정상 흐름, 실패 흐름 1개 이상, 저장소 특화 예외 케이스를 포함

테스트 프레임워크를 도입하면 실행 명령어를 이 문서에 명시하세요.

## 커밋 및 Pull Request 규칙
최근 커밋 이력은 `init`, `new skills`, `변경사항`처럼 짧은 제목을 사용하고 있습니다. 앞으로도 짧고, 명령형이며, 변경 내용을 구체적으로 드러내는 메시지를 권장합니다.

- 좋음: `Add deployment health check script`
- 피할 것: `changes`, `fix`

Pull Request에는 변경 요약, 테스트 결과, 관련 이슈를 포함하세요. UI 변경이 있을 때만 스크린샷을 첨부하면 됩니다.

## 보안 및 설정 주의사항
비밀값, 토큰, 장비별 설정 파일은 커밋하지 마세요. 로컬 설정은 `.env.local` 같은 무시 파일에 두고, 필요하면 민감 정보를 제거한 예제 파일을 별도로 제공하세요.
